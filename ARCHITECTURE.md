# بنية وتصميم النظام (CURE Operations Portal - System Architecture)

يصف هذا الملف البنية البرمجية والمكونات التقنية لمشروع **CURE Operations Portal**، بما في ذلك التقنيات المستخدمة، هيكلية المجلدات، وتدفق البيانات، والقرارات التقنية الرئيسية المتخذة أثناء التطوير.

---

## 1. التقنيات المستخدمة (Tech Stack)

تم اختيار وتطبيق التقنيات التالية بناءً على متطلبات الأداء، الأمان، والتفاعلية في الوقت الفعلي:

- **Next.js 16.2.9 (App Router)**: كإطار عمل أساسي للتطبيق بالاعتماد على ميزة Routing الحديثة والـ Server Components لتحسين الأداء وتسهيل إدارة الحالة والمسارات.
- **TypeScript (Strict Mode)**: لضمان أمان الأنواع (Type-Safety) وتقليل الأخطاء البرمجية أثناء وقت التطوير والتشغيل.
- **PostgreSQL + Prisma ORM 7.8.0**: قاعدة بيانات PostgreSQL لإدارة البيانات وهيكلتها، مع استخدام Prisma كـ ORM يوفر Type-Safety كامل وتسهيل عمليات الـ Migrations.
- **Socket.io 4.8.3 (Real-time)**: لإتاحة التحديثات الفورية ثنائية الاتجاه (مثل حالة الاتصال للممرضين وتحديثات لوحة الـ Kanban وتغييرات الـ Dispatches).
- **Tailwind CSS v4 + shadcn/ui + lucide-react**: للتصميم وبناء واجهة مستخدم متجاوبة، حديثة، وذات طابع احترافي مع استخدام المكونات الجاهزة القابلة للتخصيص.
- **@dnd-kit/core**: مكتبة مرنة وخفيفة لتوفير ميزة السحب والإفلات (Drag and Drop) داخل لوحة إدارة المهام (Kanban Board).
- **Recharts 3.8.1**: لرسم الإحصائيات والرسوم البيانية التفاعلية لتحليل الزيارات والمهام.
- **bcryptjs + jsonwebtoken (Auth)**: لتأمين عملية تسجيل الدخول وتشفير كلمات المرور، وإدارة الجلسات باستخدام الـ JWT وتوزيع الصلاحيات بناءً على الأدوار (`ADMIN`, `NURSE`, `DISPATCHER`).

## 3. تدفق البيانات (Data Flow)

يوضح المخطط التالي كيفية تدفق البيانات بين المتصفح والخوادم في حالتين: الطلبات العادية عبر الـ API، والاتصال الفوري ثنائي الاتجاه عبر الـ Socket.io.

```text
[ اتصالات الطلبات والاستعلامات العادية - HTTP Request/Response ]

  ┌─────────┐         ┌──────────────────────┐         ┌────────────┐         ┌────────────┐         ┌────────────┐
  │ Browser │ ──────> │ Next.js App Router   │ ──────> │ API Routes │ ──────> │ Prisma ORM │ ──────> │ PostgreSQL │
  │ (Client)│ <────── │ (Server Components)  │ <────── │  (Backend) │ <────── │ (Database) │ <────── │ (Database) │
  └─────────┘         └──────────────────────┘         └────────────┘         └────────────┘         └────────────┘

                                                 * * *

[ اتصالات البث الفوري والتحديثات الحية - WebSockets ]

                       ┌───────────────────────────┐
                       │     Socket.io Server      │
                       │ (Standalone Server @3001) │
                       └───────────────────────────┘
                                ▲         │
                    (Broadcast) │         │ (Emit Event / Subscribe)
                                │         ▼
                       ┌───────────────────────────┐
                       │     Browser (Client)      │
                       │   (Hook: useSocket)       │
                       └───────────────────────────┘
```

## 4. قرار تقني: لماذا استخدام Socket.io بدلاً من Server-Sent Events (SSE)؟

تم اتخاذ قرار باستخدام **Socket.io** كبديل لـ **SSE (Server-Sent Events)** للأسباب التالية:

1. **دعم الأحداث ثنائية الاتجاه (Bidirectional Communication)**: 
   تتطلب منصة العمليات إرسال واستقبال الأحداث من وإلى العميل والخادم في الوقت ذاته. على سبيل المثال، يقوم المنسق (Dispatcher) بنقل بطاقة المهمة في الكانبان (من العميل إلى الخادم)، ويقوم الخادم ببث هذا التغيير فوراً لجميع المنسقين والممرضين المتصلين في نفس اللحظة. في حين أن SSE تدعم الاتصال أحادي الاتجاه فقط (من الخادم للعميل)، فإن Socket.io توفر اتصالاُ ثنائياً متكاملاً عبر بروتوكول WebSockets.
2. **سهولة وسرعة التثبيت والإعداد**:
   يوفر Socket.io حلولاً جاهزة لـ Auto-reconnection (إعادة الاتصال التلقائي عند الانقطاع)، وإدارة الغرف (Rooms)، والبث لجميع المتصلين (Broadcasting) بشكل مدمج، مما يقلل بشكل كبير من كمية الأكواد البرمجية المطلوبة لإنشاء نظام حضور وغياب حي للممرضين ومتابعة المهام مقارنة ببنائها يدوياً باستخدام SSE.
3. **التوافق والاستمرارية**:
   تتميز مكتبة Socket.io بقدرتها على التراجع التلقائي إلى تقنية HTTP Long-Polling في البيئات والشبكات التي قد تحظر بروتوكول WebSocket، مما يضمن عمل التطبيق واستمراريته تحت أي ظرف من ظروف الاتصال.

## 5. قرار تقني: لماذا استخدام Prisma ORM؟

تم اختيار **Prisma ORM** كأداة للتعامل مع قاعدة بيانات PostgreSQL للأسباب التالية:

1. **أمان الأنواع الكامل (Full Type-Safety)**:
   توفر Prisma ميزة التوليد التلقائي لملفات TypeScript بناءً على مخطط قاعدة البيانات (`schema.prisma`). يضمن هذا بقاء الاستعلامات متوافقة تماماً مع بنية الجداول، ويوفر ميزة الإكمال التلقائي في بيئة التطوير (Auto-completion)، ويكتشف أي أخطاء في أنواع البيانات في وقت الترجمة (Compile-time) بدلاً من وقت التشغيل (Runtime).
2. **سهولة إدارة الهجرة وإجراء التغييرات (Database Migrations)**:
   يتيح Prisma Migrate أداة قوية وسهلة لتحديث قاعدة البيانات وتتبع التغييرات عبر التاريخ. بفضل أمر بسيط مثل `npx prisma db push` أو `prisma migrate dev`، يتم تحديث الجداول والعلاقات بدقة متناهية دون الحاجة لكتابة استعلامات SQL معقدة يدوياً.
3. **سهولة القراءة والصيانة (Developer Experience)**:
   تمتلك Prisma طريقة كتابة واضحة ومقروءة للاستعلامات والعلاقات (مثل استخدام `include` و `select` لجلب البيانات المرتبطة كجدول الـ AuditLog والـ User)، مما يقلل وقت التطوير ويسهل صيانة الكود من قبل أي مهندس برمجي ينضم للمشروع مستقبلاً.
